# Figma 数据提取指南

> 本文档指导如何从 Figma 文件中提取设计规范所需的数据。
> 设计规范生成器将按此流程获取并组织数据。

---

## Figma 链接识别与提取策略

> 用户输入的链接可能是项目链接、文件链接、页面链接或画板链接，必须先识别类型再决定提取范围。

> **关键规则**：Figma 中**不存在 `page-id` 参数**。页面、画板、图层等所有节点统一通过 `node-id` 引用。区分页面级和画板级的关键在于 **`t` 参数末尾数字**。

### 链接类型识别

| 类型 | URL 特征 | `t` 参数末尾 | 示例 |
|------|----------|-------------|------|
| **项目链接** | 路径含 `/files/project/` | 任意 | `figma.com/files/project/12345/...` |
| **文件链接** | 路径含 `/design/{file-key}/` 或 `/file/{file-key}/`，无 `node-id` 参数 | 任意 | `figma.com/design/ABC123/MyFile?m=auto&t=xxx-6` |
| **页面链接** | 含 `node-id=` 参数 | `-1` | `figma.com/design/ABC123/MyFile?node-id=14463:84982&t=xxx-1` |
| **画板/节点链接** | 含 `node-id=` 参数 | `-4` | `figma.com/design/ABC123/MyFile?node-id=10:20&t=xxx-4` |

### 识别决策树

```
用户输入链接
       │
       ▼
  路径含 /files/project/？
       │
  ┌────┴────┐
  Yes       No
  │         │
  ▼         ▼
 项目链接   路径含 /design/ 或 /file/？
  │              │
  │         ┌────┴────────────────┐
  │         Yes                   No → 非法链接，提示用户
  │         │
  │         ▼
  │    有 node-id 参数？
  │         │
  │    ┌────┴──────────────────┐
  │    Yes                    No
  │    │                      │
  │    ▼                      ▼
  │ 查 t 参数末尾数字？      文件链接
  │    │
  │ ┌──┴──┐
  │ -1   -4
  │ │     │
  │ ▼     ▼
  │页面   画板
  │链接   链接
  │
  ▼
  获取项目下所有文件列表
  让用户选择要生成规范的文件
```

**手动覆盖规则**：

| 场景 | 识别结果 | 说明 |
|------|----------|------|
| 画板链接 + 说"生成整个项目" | 文件链接 | 用户语言指令放大范围 |
| 页面链接 + 说"只看这个画板" | 画板链接 | 用户语言指令缩小范围 |
| node-id 链接 + 说"生成项目规范" | 文件链接 | 用户语言指令放大范围 |
| 仅截图，无链接 | 根据语言判断 | 用户描述决定范围 |

> **层级识别优先级**：用户语言指令 > Figma URL `t` 参数 > 默认文件链接

### 各类型提取策略

| 链接类型 | 可获取的数据范围 | 提取策略 |
|----------|------------------|----------|
| **项目链接** | 项目下所有文件的列表和基本信息 | 先列出所有文件，让用户选择目标文件，再按文件链接策略处理 |
| **文件链接** | 全部页面、画板、Style、组件、Token | 自动扫描全部页面，按"多页面处理策略"执行 |
| **页面链接** | 指定页面及其画板，Style 面板仍为文件级全量 | 以指定页面为主，结合文件级 Style 数据提取规范 |
| **画板链接** | 指定画板及其子节点，Style 面板仍为文件级全量 | 从单个画板提取局部数据，**仅适用于补充/更新单个组件的规范**，不推荐用于生成完整项目规范 |

### 特殊场景处理

| 场景 | 处理方式 |
|------|----------|
| 用户给了画板链接但要求"生成完整规范" | 告知用户画板链接只能覆盖局部，建议提供文件链接；如用户坚持，仅基于该画板 + Style 面板生成，在文档中标注"数据来源：单个画板，可能不完整" |
| 用户给了项目链接 | 列出项目下所有文件，建议用户选择目标文件 |
| 用户给了多个链接 | 合并处理：以文件链接为主要数据源，页面/画板链接作为补充验证 |
| 链接无法访问 | 降级为截图模式，参照 SKILL.md 降级模式处理 |

### 链接识别后的确认话术

```
🔗 链接分析结果：

链接类型：[文件链接 / 页面链接 / 画板链接 / 项目链接]
目标范围：[整个文件 / 指定页面"XXX" / 指定画板"XXX" / 项目"XXX"下所有文件]

📋 推荐提取范围：
[根据链接类型给出建议，如：
- 文件链接 → "建议扫描全部页面（组件库 + 最新迭代页）"
- 页面链接 → "将以指定页面为主，结合文件级样式数据"
- 画板链接 → "⚠️ 单画板数据有限，建议改为文件链接以生成完整规范"
]

是否继续？或提供其他链接。
```

## 多页面与多画板处理策略

> 实际项目中一个 Figma 文件通常包含多个页面、每个页面多个画板。本节定义如何从复杂文件结构中正确提取规范数据。

### 页面分类与优先级

提取数据时，不同类型的页面拥有不同的数据可信度：

| 页面类型 | 识别方式 | 优先级 | 说明 |
|----------|----------|--------|------|
| **组件库页** | 名称含"组件""Components""📦" | ⭐⭐⭐ 最高 | 组件定义、变体、Token 的权威来源 |
| **设计规范页** | 名称含"规范""Tokens""Design System""📐" | ⭐⭐⭐ 最高 | Token 定义、样式说明文档 |
| **最新迭代页** | 按日期排序的最新设计页面 | ⭐⭐ 中 | 反映当前最新的设计决策 |
| **早期迭代页** | 较早的版本迭代页面 | ⭐ 低 | 可能包含已被废弃的旧设计 |

> **核心原则：Token 定义（Style 面板） > 组件库引用 > 画板实际使用。**
> 样式面板中定义的值是"规范应该是什么"，画板中实际使用的值是"规范是否被遵守"——两者可能不一致，需要分别记录。

### 画板扫描策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **组件库优先** | 只扫描组件库页中的主组件和变体 | 提取 M08 组件规范时 |
| **Style 面板优先** | 只看 Color/Text/Effect Style 面板定义 | 提取 M03 色彩、M04 排版时 |
| **最新画板抽样** | 在最新迭代页中抽取 3-5 个代表性画板 | 验证规范实际执行情况 |
| **全量扫描** | 扫描所有页面的所有画板 | 发现规范违规和不一致（需结合 design-file-audit） |

### 数据聚合方法

当多个页面/画板中出现相同类型的数据时，按以下方法聚合：

#### 色彩数据

```
1. 先从 Color Style 面板提取全部样式定义 → 作为"规范值"
2. 再扫描全部画板中的实际填充色 → 统计使用频率
3. 对比规范值与实际值：
   ├── 一致 → 直接纳入规范文档
   └── 不一致（存在裸色值或未使用 Style）→ 列入"一致性问题"清单
```

#### 排版数据

```
1. 先从 Text Style 面板提取全部样式定义 → 作为"规范值"
2. 再扫描全部画板中的文字图层 → 统计字号/字重/行高分布
3. 频率分析：
   ├── 高频使用的字号 → 确认为规范梯度中的层级
   └── 低频/一次性使用的字号 → 标注为特殊情况或异常值
```

#### 间距数据

```
1. 统计所有 Auto Layout 的 gap 和 padding 值
2. 按值的出现频率排序
3. 频率 TOP 5-8 的值 → 确认为间距梯度
4. 其余值 → 标注为"非标准间距"，列出出现位置
```

#### 组件数据

```
1. 从组件库页提取全部主组件及变体定义
2. 统计每个组件在全部画板中的实例数量
3. 按实例数量降序排列 → 确定组件优先级
4. 实例数量 ≥ 3 的组件 → 纳入规范文档
5. 实例数量 < 3 的组件 → 标注为"低频组件"，可选纳入
```

### 跨页面冲突处理

当不同页面中出现不一致的数据时：

| 冲突类型 | 处理方式 |
|----------|----------|
| 同一 Token 在 Style 面板只有一个定义，但画板中有不同值 | 以 Style 面板为准，画板偏差列为"一致性问题" |
| Style 面板中存在重复定义（同色值同语义多个 Style） | 列入"重复 Token"问题，标注所有重复项 |
| 新旧迭代页使用了不同配色/字号 | 以最新迭代页为准，标注历史变更 |
| 组件库中的组件与画板中的独立元素视觉一致 | 以组件库为准，建议将画板中的独立元素替换为组件实例 |

### 输出中的冲突报告

在生成规范文档时，如果发现跨页面不一致，在附录中输出冲突报告：

```markdown
## 一致性问题清单

| # | 问题类型 | 规范值 | 实际值 | 出现位置 | 建议 |
|---|----------|--------|--------|----------|------|
| 1 | 裸色值 | color/brand/primary (#1677FF) | #1677FF（未关联 Style） | V2.1 详情页 / 3处 | 关联 Color Style |
| 2 | 非标准间距 | space-md (12px) | 10px | V2.0 列表页 / 按钮组 | 调整为 12px |
| 3 | 裸文字样式 | font/body-md (14px/Regular) | 14px/Regular（未关联） | V2.2 搜索页 / 标签 | 关联 Text Style |
```

## Figma Variables 获取与匹配

> 在 Step 0.5 执行。优先使用文件中已定义的 Variables 作为规范基准，未定义为 Variable 的高频色值提示补充。

### API 端点

| 端点 | 说明 | 认证方式 |
|------|------|----------|
| `GET /v1/files/{key}/variables/local` | 获取本地变量（草稿态，未发布） | `X-Figma-Token: {token}` 请求头 |
| `GET /v1/files/{key}/variables/published` | 获取已发布的变量 | 同上，或 URL 参数 `?token={token}` |

**推荐调用顺序**：先调 `/variables/local`（设计进行中更可能遇到），再调 `/variables/published`（已发布到 Team Library 的变量）。

### 响应结构（关键字段）

```json
{
  "meta": {
    "variables": {
      "VariableID:1:2": {
        "id": "VariableID:1:2",
        "name": "brand/primary",
        "value": { "r": 0.149, "g": 0.427, "b": 1.0, "a": 1.0 },
        "type": "COLOR",
        "scopes": ["FRAME_FILL", "SHAPE_FILL", "TEXT_FILL"],
        "codeSyntax": { "WEB": "--brand-primary" }
      },
      "VariableID:1:3": {
        "id": "VariableID:1:3",
        "name": "semantic/error",
        "value": "VariableID:1:2",
        "type": "COLOR",
        "scopes": ["FRAME_FILL", "SHAPE_FILL"]
      }
    },
    "variableCollections": {
      "VariableCollectionID:1": {
        "id": "VariableCollectionID:1",
        "name": "Global Colors",
        "modes": [{ "modeId": "1", "name": "Default" }]
      }
    }
  }
}
```

### 变量别名解析

当 `value` 字段是字符串（以 `"VariableID:` 开头）时，表示引用其他变量，需递归解析：

```
解析函数 resolve_value(variable_id, variables_map):
  1. 从 variables_map 中取出该 variable 的 value
  2. 如果 value 是对象（含 r/g/b/a）→ 直接返回 RGBA
  3. 如果 value 是字符串且以 "VariableID:" 开头
     → 提取被引用的 variable_id，递归调用 resolve_value
  4. 返回最终的 RGBA 值
```

**示例**：
- `VariableID:1:3` 的 `value` = `"VariableID:1:2"`（引用 primary）
- `VariableID:1:2` 的 `value` = `{"r":0.149, "g":0.427, "b":1.0, "a":1.0}`
- 解析结果：`VariableID:1:3` 的最终色值 = `#266DFF`

### HEX → Variable 查找表构建

对每个 `type == "COLOR"` 的变量，解析最终色值后建立映射：

```
对每个 Variable（type == "COLOR"）：
  1. 调用 resolve_value(variable.id, variables_map) 获取最终 RGBA
  2. RGBA(float 0-1) → HEX：
     hex = "#" + hex(round(r*255)) + hex(round(g*255)) + hex(round(b*255))
     （注意：alpha 通道不参与 HEX 匹配，仅用于不透明度记录）
  3. 存入查找表：{ hex_value: variable_name }
```

**示例查找表**：
```json
{
  "#266DFF": "brand/primary",
  "#F5222D": "semantic/error",
  "#18E645": "semantic/success",
  "#292F36": "neutral/bg-page"
}
```

### 与 Step 2 归类的集成

| 场景 | Step 2 归类处理方式 |
|------|----------|
| 色值精确匹配到 Variable | Token 名称使用 Variable 名称（如 `brand/primary`），来源标注 `"Figma Variable"` |
| 色值未匹配到 Variable，但出现频次 ≥ 10 | 归类为"待补充为 Variable"，记录到附录清单 |
| Style Panel 有值且 Variable 也有 | Variable 优先，Style 值作为交叉验证，如不一致标注警告 |
| 文件中无 Variable（两个 API 均返回空 `meta.variables`） | 按原有逻辑处理，附录标注"未检测到 Figma Variables" |

### 待补充为 Variable 的判断标准

扫描完成后，以下情况列入附录"待补充为 Variable 的颜色清单"：

| 条件 | 说明 |
|------|------|
| 色值在文件中出现频次 ≥ 10 次 | 高频使用，值得定义为 Variable |
| 色值未匹配到任何 Variable | 目前是裸色值，无统一管理 |
| 色值未关联任何 Color Style | 连 Style 都没有，更需 Variable 统一管理 |
| （可选）色值已关联 Color Style 但未定义 Variable | 建议将 Style 升级为 Variable |

## 颜色冲突检测

> 在数据归类完成后、生成文档前执行。对已归类的色彩 Token 进行两两比对，自动发现重复色、近似色和疑似冲突，让用户确认后再写入最终文档。

### 检测时机与范围

| 属性 | 说明 |
|------|------|
| **执行时机** | Step 2（数据归类）完成后，Step 3（缺口识别）之前 |
| **检测范围** | Step 2 归类到 M03 色彩系统的所有带语义名称的 Token（品牌色、语义色、中性色、标签色等） |
| **不检测** | 原始全量色值（数万个节点填充色）、带透明度的色值（alpha < 1）、渐变色 |

### 三类冲突定义

| 冲突类型 | 判定条件 | 说明 | 建议动作 |
|----------|----------|------|----------|
| **完全重复** | HEX 色值完全相同，但被两个或以上不同 Token 引用 | 可能是有意共用（品牌色 = 预警色），也可能是命名冗余 | 确认语义；如果是冗余则合并 |
| **近似色** | CIE76 ΔE ≤ 3 | 人眼几乎无法区分的差异 | 建议统一为一个 Token |
| **疑似冲突** | CIE76 ΔE 3~5 | 人眼可感知差异，但可能是有意设计 | 提交用户判断 |

> ΔE > 5 的色值对视为明显不同色，不标记为冲突。

### CIE76 ΔE 算法

色差计算需将 HEX 色值转换为 CIE Lab 空间后计算欧氏距离。

#### 步骤 1：HEX → RGB

```
R = int(hex[0:2], 16) / 255.0
G = int(hex[2:4], 16) / 255.0
B = int(hex[4:6], 16) / 255.0
```

#### 步骤 2：RGB → XYZ（D65 白点）

```
# Gamma 校正
R_linear = R > 0.04045 ? ((R + 0.055) / 1.055) ^ 2.4 : R / 12.92
G_linear = G > 0.04045 ? ((G + 0.055) / 1.055) ^ 2.4 : G / 12.92
B_linear = B > 0.04045 ? ((B + 0.055) / 1.055) ^ 2.4 : B / 12.92

# 线性 RGB → XYZ（sRGB 矩阵 × D65）
X = 0.4124564 × R_linear + 0.3575761 × G_linear + 0.1804375 × B_linear
Y = 0.2126729 × R_linear + 0.7151522 × G_linear + 0.0721750 × B_linear
Z = 0.0193339 × R_linear + 0.1191920 × G_linear + 0.9503041 × B_linear
```

#### 步骤 3：XYZ → CIE Lab

```
# D65 白点参考值
Xn = 0.95047, Yn = 1.00000, Zn = 1.08883

# f 函数
f(t) = t > (6/29)^3 ? t^(1/3) : t / (3 × (6/29)^2) + 4/29

L = 116 × f(Y/Yn) - 16
a = 500 × (f(X/Xn) - f(Y/Yn))
b = 200 × (f(Y/Yn) - f(Z/Zn))
```

#### 步骤 4：计算 ΔE

```
ΔE = √[(L1 - L2)² + (a1 - a2)² + (b1 - b2)²]
```

### ΔE 阈值参考

| ΔE 范围 | 人眼感知 | 处理建议 |
|---------|----------|----------|
| 0 | 完全相同 | 完全重复 |
| 0 < ΔE ≤ 1 | 几乎不可察觉 | 强烈建议合并 |
| 1 < ΔE ≤ 3 | 仅在并排对比时可区分 | 建议合并 |
| 3 < ΔE ≤ 5 | 普通人可感知差异 | 提交用户判断 |
| ΔE > 5 | 明显不同色 | 正常，不标记 |

### 排除规则

以下情况不纳入冲突检测：

| 排除项 | 原因 |
|--------|------|
| 带 alpha 的色值（opacity < 1） | 色差可能来自透明度而非底色 |
| 渐变色 | 多色渐变无法用单一色差衡量 |
| 同一语义的状态变体（如 primary / primary-hover / primary-active） | 状态色本就应有差异 |
| 图标填充色（未关联 Style） | 非规范级 Token |
| 装饰性元素色值 | 非规范级 Token |

### 检测执行方式

AI 在执行此步骤时，可选择以下方式之一：

1. **心算估算**（少量 Token，≤ 30 个）：AI 直接对比色值的 RGB 分量差异，估算 ΔE 级别。适用于 Token 数量较少的场景。
2. **辅助脚本计算**（大量 Token，> 30 个）：生成临时 Python 脚本执行精确计算，确保结果准确。脚本需包含完整的 HEX → Lab 转换和 ΔE 计算逻辑。

> 无论使用哪种方式，最终结果都应按上方"三类冲突定义"的表格格式呈现给用户确认。

## 数据提取流程

```
用户提供 Figma 链接
       │
       ▼
 使用 figma skill 获取文件数据
       │
       ├── 获取文件结构（页面、画板层级）
       ├── 获取样式数据（Color/Text/Effect Styles）
       ├── 获取组件数据（Components & Variants）
       └── 获取设计Token（Spacing/Radius/Opacity 等）
       │
       ▼
 分析与归类
       │
       ├── 色彩 → 归入 M03
       ├── 排版 → 归入 M04
       ├── 间距/圆角 → 归入 M05
       ├── 组件 → 归入 M08
       └── 布局/网格 → 归入 M07
       │
       ▼
 按模块化模板填充输出
```

## 各模块数据提取方法

### M03 色彩系统

**提取重点**：

| 数据源 | 提取内容 | 方法 |
|--------|----------|------|
| Color Styles | 所有颜色样式名称、色值 | figma skill 样式面板数据 |
| 画板填充 | 背景色、容器色 | 扫描 Frame 的 fills 属性 |
| 组件样式 | 按钮、标签等的颜色映射 | 分析组件变体的颜色差异 |

**分析逻辑**：
1. 按 Style 名称前缀归类（brand / semantic / neutral / dark）
2. 识别同色值不同语义的 Token（允许）和同色值同语义的重复（违规提示）
3. 统计使用频率，标记高频色和低频色

### M04 排版系统

**提取重点**：

| 数据源 | 提取内容 | 方法 |
|--------|----------|------|
| Text Styles | 所有文字样式名称、字体、字号、行高、字重 | figma skill 样式面板数据 |
| 文字图层 | 实际使用的字体和字号 | 扫描 Text 节点 |
| 组件内文字 | 按钮、输入框等组件的字号 | 分析组件内文字节点 |

**分析逻辑**：
1. 提取所有 Text Style，按名称归类
2. 统计字号使用频率，识别字号梯度
3. 检查是否存在非标准字号（如 13px、17px、21px 等非偶数）
4. 识别裸文字样式（未关联 Text Style 的文字节点）

### M05 间距与圆角

**提取重点**：

| 数据源 | 提取内容 | 方法 |
|--------|----------|------|
| Auto Layout | gap、padding、itemSpacing | 扫描 Auto Layout 节点属性 |
| Frame 间距 | 元素间的绝对间距 | 分析兄弟节点的位置差 |
| 圆角值 | cornerRadius 属性 | 扫描所有带圆角的节点 |

**分析逻辑**：
1. 统计所有间距值，识别间距梯度（是否为4的倍数）
2. 统计所有圆角值，识别圆角规范
3. 标记非标准值（如 5px、7px、9px 等非4倍数）

### M06 图标系统

**提取重点**：

| 数据源 | 提取内容 | 方法 |
|--------|----------|------|
| 图标组件 | 图标尺寸、线宽、风格 | 扫描 Icon 相关组件 |
| 图标图层 | 实际使用的尺寸和颜色 | 扫描布尔运算节点 |
| 图标集 | 使用的图标库 | 检查组件名称前缀 |

### M07 布局与网格

**提取重点**：

| 数据源 | 提取内容 | 方法 |
|--------|----------|------|
| 布局网格 | 列数、槽宽、边距 | 分析画板的 layoutGrids 属性 |
| 约束条件 | 元素的固定/弹性设置 | 分析 constraints 属性 |
| 画板尺寸 | 常见画板宽度（推断断点） | 统计 Frame 宽度分布 |

### M08 组件规范

**提取重点**：

| 数据源 | 提取内容 | 方法 |
|--------|----------|------|
| 组件定义 | 变体属性、默认状态 | figma skill 组件面板数据 |
| 组件尺寸 | 宽高、内边距 | 分析组件的 boundingBox 和 Auto Layout |
| 组件颜色 | 各变体的颜色映射 | 分析组件变体的样式差异 |
| 组件文字 | 按钮文案字号、字重 | 分析组件内文字节点 |

**分析逻辑**：
1. 按组件名称前缀归类（Button / Input / Card / Modal 等）
2. 对每个组件提取变体矩阵（Variant × State × Size）
3. 记录组件的尺寸、颜色、间距等具体数值
4. 识别组件的使用频率（实例数量）

## 数据不足时的处理

| 情况 | 处理方式 |
|------|----------|
| 某模块数据稀少 | 标注"数据不足"，基于可见内容推断，标注"[推断]" |
| 无法访问 Figma API | 降级为截图分析模式，基于可视内容提取 |
| 样式面板为空 | 建议用户先建立设计Token，再生成规范 |
| 组件未使用 Variant | 标注"建议升级为变体组件"，记录当前独立组件信息 |

## 质量校验

提取完成后自动执行以下校验：

1. **色值格式**：所有色值必须为 `#XXXXXX` 格式
2. **数值格式**：间距/圆角必须标注单位（px）
3. **Token 命名**：检查是否符合团队命名体系
4. **数据完整性**：必需模块是否都有足够数据支撑
5. **一致性**：同一参数在不同位置引用时是否一致
