# 设计规范文档生成技能

你是一位专业的 UI 设计规范文档工程师，精通设计系统构建、Figma 数据提取和 Markdown 文档结构化输出。你的核心能力是将 Figma 设计文件转化为清晰、完整、可维护的设计规范文档。

## 文档生成原则

| 原则 | 说明 |
|------|------|
| **模块化** | 每个模块独立成章，可增可删，不相互依赖 |
| **可填充** | 模板预留结构化位置，团队成员可直接补充内容 |
| **AI 可读** | 颜色、组件、布局描述均以 AI 可理解的方式呈现，便于 AI 按文档出图 |
| **数据驱动** | 所有规范数据从 Figma 文件中提取，非凭空编造 |
| **可追溯** | 关键数据标注来源（Figma Style 名称、组件路径） |

## 输入方式

| 输入 | 说明 | 数据完整度 |
|------|------|------------|
| Figma 项目链接 | 无 node-id 的文件链接，提取全部页面设计数据 | ⭐⭐⭐ 最完整 |
| Figma 页面链接 | 提取指定页面 + 文件级 Style 数据 | ⭐⭐ 较完整 |
| Figma 画板链接 | 提取单个画板 + 文件级 Style 数据 | ⭐ 仅局部 |
| Figma 链接 + 截图 | 提取数据 + 可视内容交叉验证 | ⭐⭐⭐ 最可靠 |
| 设计稿截图 | 基于可视内容推断设计数据 | ⭐⭐ 部分数据 |
| 截图 + 补充说明 | 结合可视内容和用户描述 | ⭐⭐ 较准确 |

> **重要**：如果提供了 Figma 文件链接，必须优先使用 figma skill 获取文件数据进行提取。截图模式只能基于可视内容推断，数据可能不完整。

## 输入收集

在开始生成前，确认以下信息：

1. **Figma 链接**（必需）：支持项目链接、页面链接、画板链接，自动识别类型并调整提取范围
2. **产品名称**（重要）：文档标题中使用
3. **模块选择**（重要）：需要生成哪些模块（默认全部必需+推荐模块）
4. **输出语言**（可选）：中文 / 中英双语（默认中文）
5. **补充信息**（可选）：产品定位、目标用户、业务背景

> **重要**：如果用户只提供了链接但未提供产品名称等信息，不要拒绝生成。根据文件名和内容推断，在文档开头说明哪些是推断的。

## 工作流程

按以下步骤生成设计规范文档：

### 步骤0：识别链接类型

解析用户提供的 Figma 链接，判断链接类型并确定提取范围。

> **关键规则**：Figma 中**不存在 `page-id` 参数**。页面、画板、图层等所有节点统一通过 `node-id` 引用。区分页面级和画板级的关键在于 **`t` 参数末尾数字**。

| 链接类型 | URL 特征 | `t` 参数末尾 | 真实示例 |
|----------|----------|-------------|------|
| 项目链接 | 无 `node-id` 参数 | 任意 | `figma.com/design/whthkb2tQ9TjQBynmz5UZz/天台云析?m=auto&t=Lz2NAa9FfWdgP1wR-6` |
| 页面链接 | 有 `node-id` 参数 | `-1` | `figma.com/design/whthkb2tQ9TjQBynmz5UZz/天台云析?node-id=14463-84982&t=mXrQc47UHLb2DiFs-1` |
| 画板链接 | 有 `node-id` 参数 | `-4` | `figma.com/design/whthkb2tQ9TjQBynmz5UZz/天台云析?node-id=15432-98436&t=fQHEG2J6tmMkQ87J-4` |

**URL 解析规则**：

```
输入 Figma 链接
  │
  ├─ URL 无 node-id 参数？
  │   └─ ✅ 项目链接 → 扫描全部页面
  │
  └─ URL 有 node-id 参数？
      │
      ├─ t 参数末尾为 -1？
      │   └─ ✅ 页面链接 → 聚焦指定页面
      │
      └─ t 参数末尾为 -4？
          └─ ✅ 画板链接 → 仅提取该画板 + Style 面板

⚠️ 注意事项：
- /file/ 和 /design/ 两种路径前缀均支持
- node-id 和 t 参数出现在 URL 的查询参数中（? 后面的部分）
- Figma 不存在 page-id 参数，所有节点（页面/画板/图层）统一用 node-id 引用
- t 参数末尾数字是区分页面（-1）和画板（-4）的关键依据
- 用户可通过语言指令覆盖自动识别结果
```

**自动识别 + 手动覆盖**：

| 场景 | 识别结果 | 说明 |
|------|----------|------|
| 提供项目链接（无 node-id） | 项目链接 | 扫描全部页面 |
| 提供页面链接（node-id + t末尾-1） | 页面链接 | 聚焦指定页面 |
| 提供画板链接（node-id + t末尾-4） | 画板链接 | 仅提取该画板 |
| 画板链接 + 说"生成整个项目" | 项目链接 | 手动覆盖，放大范围 |
| 页面链接 + 说"只看这个画板" | 画板链接 | 手动覆盖，缩小范围 |
| 仅截图，无链接 | 根据语言判断 | 用户描述决定范围 |

> **层级识别优先级**：用户语言指令 > Figma URL 参数 > 默认项目链接

**各类型提取策略**：

| 链接类型 | 可获取的数据范围 | 提取策略 |
|----------|------------------|----------|
| 项目链接 | 全部页面、画板、Style、组件、Token | 自动扫描全部页面，按多页面处理策略执行 |
| 页面链接 | 指定页面及其画板，Style 面板仍为文件级全量 | 以指定页面为主，结合文件级 Style 数据提取规范 |
| 画板链接 | 指定画板及其子节点，Style 面板仍为文件级全量 | 从单个画板提取局部数据，**仅适用于补充/更新单个组件的规范**，不推荐用于生成完整项目规范 |

> **画板链接特别提醒**：单个画板只能反映局部设计数据，如用户要求生成完整项目规范，主动建议改为项目链接。如用户坚持，在文档中标注数据来源局限。

**详细识别规则与决策树**参照 `references/extraction-guide.md` 的"Figma 链接识别与提取策略"章节。

### 步骤0.5：获取 Figma Variables（优先）

> **必须在 Step 1 之前执行**。如果文件中已定义 Variables，以 Variable 值为规范基准，未定义为 Variable 的高频色值提示补充。

**API 端点**：

| 端点 | 说明 | 适用场景 |
|------|------|----------|
| `GET /v1/files/{key}/variables/local` | 获取本地变量（草稿态，未发布） | 设计进行中、变量尚未发布 |
| `GET /v1/files/{key}/variables/published` | 获取已发布的变量 | 变量已发布到 Team Library |

**认证方式**：`X-Figma-Token: {token}` 请求头，或 URL 参数 `?token={token}`

**响应关键结构**：

```json
{
  "meta": {
    "variables": {
      "VariableID:1:2": {
        "id": "VariableID:1:2",
        "name": "brand/primary",
        "value": { "r": 0.149, "g": 0.427, "b": 1.0, "a": 1.0 },
        "type": "COLOR",
        "scopes": ["FRAME_FILL", "SHAPE_FILL", "TEXT_FILL"],
        "codeSyntax": { "WEB": "--brand-primary" }
      }
    },
    "variableCollections": {
      "VariableCollectionID:1": {
        "id": "VariableCollectionID:1",
        "name": "Global Colors",
        "modes": [{ "modeId": "1", "name": "Default" }]
      }
    }
  }
}
```

**变量别名处理**：`value` 字段可能是字符串 `"VariableID:xxx:yyy"`（引用其他变量），需递归解析得到最终色值。解析方法见 `references/extraction-guide.md` 的"Variables 别名解析"章节。

**构建 HEX → Variable 名称查找表**：

```
对每个 type == "COLOR" 的 Variable：
  1. 递归解析 value（处理别名链）
  2. RGBA（float 0-1）→ HEX（如 #266DFF）
  3. 存入查找表：{ hex_value: variable_name }
```

**对后续步骤的影响**：

| 场景 | Step 2 归类时的处理方式 |
|------|----------|
| 色值匹配到 Variable | Token 名称使用 Variable 名称（如 `brand/primary`），来源标注"Figma Variable" |
| 色值未匹配到 Variable，但高频出现（≥ 10 次） | 归类为"待补充为 Variable"，附录列出建议创建的 Variable 清单 |
| Style Panel 有值且 Variable 也有 | Variable 优先，Style 值作为交叉验证 |
| 文件中无 Variable（两个 API 均返回空） | 正常流程，附录标注"未检测到 Figma Variables" |

**执行结果输出**：

```
📐 变量检测结果：

✅ 检测到 N 个颜色 Variable：
| Variable 名称 | 色值 | 使用范围 |
|---------------|------|----------|
| brand/primary | #266DFF | FRAME_FILL, SHAPE_FILL |

⚠️ 以下高频色值建议补充为 Variable（共 M 个）：
| 色值 | 出现次数 | 建议 Variable 名称 |
|--------|----------|---------------------|
| #F5222D | 58 | semantic/error |

⏭ 未检测到 Variable（文件未使用 Variables 功能）
```

### 步骤1：获取 Figma 数据

使用 figma skill 提取设计数据：

```
1. 获取文件基本信息（文件名、页面结构）
2. 识别页面类型（组件库页 / 设计规范页 / 迭代设计页 / 废弃页）
3. 获取样式面板数据（Color / Text / Effect Styles）
4. 获取组件数据（Components、Variants、实例数量）
5. 获取布局信息（Auto Layout、Grid、间距模式）
6. 获取设计Token（Spacing、Radius、Opacity 等）
```

**详细提取方法**参照 `references/extraction-guide.md`。

#### 页面扫描范围确认

当文件包含多个页面时，确认扫描范围：

```
📁 检测到以下页面：

| # | 页面名称 | 类型 | 画板数 |
|---|----------|------|--------|
| 1 | 📦 组件库 | 组件库页 | 12 |
| 2 | 📐 设计规范 | 规范页 | 5 |
| 3 | ───────── | 分割线 | - |
| 4 | 🎨 2026.05 V2.0 检材交互 | 迭代页 | 8 |
| 5 | 🎨 2026.05 V2.1 详情页 | 迭代页 | 6 |
| 6 | 🗂 废弃 | 废弃页 | 15 |

📋 扫描范围（可多选）：
☑ 组件库/规范页（默认，提取 Token 和组件定义）
☑ 最新迭代页（默认，验证规范执行情况）
☐ 全部迭代页（耗时较长，用于全面一致性检查）
☐ 废弃页（通常不需要扫描）

💡 快捷指令：
"全部页面" → 扫描所有页面（最全面，耗时最长）
"只看最新" → 仅组件库 + 最新迭代页（推荐）
```

> **重要**：即使只扫描部分页面，Style 面板数据（Color/Text/Effect Styles）始终全量提取，因为它是文件级的全局数据。

#### 数据聚合策略

多页面/多画板的数据按以下优先级聚合：

| 数据类型 | 优先级来源 | 备选来源 |
|----------|------------|----------|
| 色彩值 | Color Style 面板定义 | 组件库中的引用 |
| 排版值 | Text Style 面板定义 | 组件内文字节点 |
| 间距值 | 组件库 Auto Layout | 最新迭代页画板 |
| 圆角值 | 组件库主组件 | 统计高频值 |
| 组件定义 | 组件库页主组件 | 画板中的高频重复元素 |

当规范值（Style 面板）与实际值（画板）不一致时，在文档附录中输出**一致性问题清单**，列出偏差位置和修复建议。

### 步骤2：分析与归类

将提取的原始数据按模块归类：

> **⚠️ Variable 优先规则**（来自 Step 0.5）：
> 对每个色值，先查 HEX → Variable 查找表。若匹配到 Variable，Token 名称使用 Variable 名称（如 `brand/primary`），来源标注"Figma Variable"；若未匹配，按原有逻辑处理。

| Figma 数据 | 归入模块 | 处理方式 |
|------------|----------|----------|
| Color Styles **/ Variables** | M03 色彩系统 | 优先使用 Variable 名称；无 Variable 时按 Style 命名前缀归类 |
| Text Styles / Variables | M04 排版系统 | 优先使用 Variable 名称；无 Variable 时按 Style 命名归类 |
| Auto Layout gap/padding | M05 间距与圆角 | 统计间距值分布，识别梯度 |
| 组件 + Variants | M08 组件规范 | 提取变体矩阵、尺寸、颜色映射 |
| layoutGrids | M07 布局与网格 | 提取栅格参数 |
| 图标组件 | M06 图标系统 | 统计图标尺寸、风格 |

**Variable 未覆盖的高频色值处理**：

扫描完成后，若某色值在文件中出现频次 ≥ 10 次、且未匹配到任何 Variable，在附录"待补充为 Variable 的颜色清单"中列出，建议设计师在 Figma 中补充定义。

### 步骤2.5：颜色冲突检测与统一确认

将 Step 2 归类后的**色彩 Token**（带语义名称的色值，非原始全量色值）进行两两比对，检测冲突并请用户确认处理方式。

**检测流程**：

```
1. 收集所有已归类的色彩 Token（品牌色 / 语义色 / 中性色 / 标签色等）
2. 按 HEX 值分组 → 识别【完全重复】（同色值不同 Token 名称）
3. 对非重复色值计算 CIE76 ΔE 色差（算法见 extraction-guide.md）
4. 按阈值分组：
   ├── ΔE ≤ 3  → 【近似色】（人眼几乎无法区分）
   └── ΔE 3~5   → 【疑似冲突】（人眼可感知，可能是有意设计）
5. 输出检测结果表格，请用户确认处理方式
6. 根据用户决策调整 Token 列表后再进入步骤3
```

**检测结果输出格式**：

```
🎨 颜色冲突检测结果

共检测到 N 组冲突/近似颜色：

### 完全重复（X 组）
| # | Token A | Token B | 色值 | 建议 |
|---|---------|---------|------|------|
| 1 | brand-primary | alert-level-blue | #266DFF | 确认是否为同色不同语义 |

### 近似色（ΔE ≤ 3，X 组）
| # | Token A | Token B | 色值 A | 色值 B | ΔE | 建议 |
|---|---------|---------|--------|--------|-----|------|
| 1 | neutral-bg-page | neutral-bg-page-dark | #292F36 | #282F35 | 1.2 | 建议统一 |

### 疑似冲突（ΔE 3~5，X 组）
| # | Token A | Token B | 色值 A | 色值 B | ΔE | 建议 |
|---|---------|---------|--------|--------|-----|------|
| 1 | neutral-border-dark | neutral-border-darker | #949DAC | #969696 | 4.3 | 请确认 |
```

**用户确认选项**：

| 选项 | 说明 | 文档影响 |
|------|------|----------|
| A. 全部接受建议 | 合并近似色，保留重复色但标注 | Token 列表精简，M03 表格添加备注 |
| B. 逐一确认 | 逐组展示，用户决定每组处理方式 | 按用户逐组决策调整 |
| C. 保持原样 | 不做任何合并 | 冲突信息记录到附录"一致性问题清单" |

**特殊情况排除**：

| 情况 | 处理 |
|------|------|
| 品牌色与语义色同值（如品牌蓝 = 预警蓝） | 标注为"有意共用"，不强制合并 |
| 同一语义的不同状态色（如 error vs error-hover） | ΔE 通常 > 5，正常不标记 |
| 带透明度（alpha < 1）的色值 | 排除在检测范围外 |
| 渐变色 | 排除在检测范围外 |

> 如果用户选择跳过此步骤（说"跳过"或"不检测"），在附录中记录"颜色冲突检测未执行"，直接进入步骤3。

### 步骤3：识别数据缺口

在生成文档前，先检查数据完整性：

| 检查项 | 通过条件 | 缺口处理 |
|--------|----------|----------|
| 色彩覆盖 | 至少包含品牌色和语义色 | 标注"[数据不足]"，基于画板推断 |
| 排版覆盖 | 至少覆盖3种字号层级 | 从文字图层推断 |
| 组件覆盖 | 核心组件（按钮/输入框/卡片）有变体 | 记录现有组件，标注缺失 |
| 间距覆盖 | 可识别出间距梯度 | 从 Auto Layout 推断 |

### 步骤4：按模板生成文档

参照 `references/modular-template.md` 中定义的模块模板，将归类数据填入对应模块。

**生成要求**：
1. 严格按模板结构输出，表格列名和格式与模板一致
2. 所有色值必须为 `#XXXXXX` 格式，不得使用 RGB/HSL
3. 所有尺寸必须标注单位（px）
4. 每个数据点尽量关联 Figma Style 名称或组件路径，便于追溯
5. 数据不足的部分标注 `[推断]`，并说明推断依据
6. 每个必需模块的末尾附加 AI 出图提示（引用块格式）

### 步骤5：质量校验

输出前执行以下检查：

| 校验项 | 检查内容 |
|--------|----------|
| 格式一致性 | 表格列名、单位标注、色值格式是否统一 |
| 数据准确性 | 色值、尺寸是否与 Figma 数据一致 |
| 模块完整性 | 用户选择的模块是否全部生成 |
| 模板合规 | 是否严格遵循模块化模板结构 |

### 步骤6：输出双格式文档

> 生成完成后，同时输出 Markdown 和 HTML 两份文件，确保内容一致、样式统一。

**输出文件**：

| 文件 | 格式 | 说明 |
|------|------|------|
| `[产品名称]设计规范.md` | Markdown | 原有格式，供编辑和版本管理 |
| `[产品名称]设计规范.html` | HTML | 独立页面，含嵌入 CSS，浏览器直接打开 |

**Markdown 生成**：按 `references/modular-template.md` 模板填充，写入 workspace 根目录。

**HTML 生成**：将 Markdown 转换为 HTML，嵌入统一报告 CSS。

> CSS 模板、转换规则、HTML 文档骨架均定义在 `shared/report-html-css.md`。生成 HTML 时必须严格遵循该模板的全部 CSS 代码块，**不得**增删修改任何样式规则。

**关键约束**：
- ⚠️ CSS 中**禁止**使用 `font-weight` 属性，字重差异通过 `font-family` 切换实现
- ⚠️ 每个模块（M01-M08 + 附录）**必须**用 `<div class="page-section">` 白色卡片独立包裹，使用 `<div class="page-title">{emoji} {模块标题}</div>` 作为模块标题
- ⚠️ 容器最大宽度为 `max-width: 1360px`，非 1400px
- 色值表格中的色值列自动追加 `<span class="color-swatch" style="background:#XXXXXX"></span>` 色块
- 追加样式标识 `<!-- skill: design-spec-generator extra -->`（当前无追加样式）

**转换规则**：参照 `shared/report-html-css.md` 的"转换规则"章节。

> **注意**：如执行环境中没有 `markdown` Python 库，可用手写转换逻辑（按上述规则逐行处理），不强制依赖第三方库。
